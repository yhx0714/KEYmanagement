# 可信数据空间密钥管理与加密演示系统：关键业务流程方案设计

## 1. 文档目标

本文档对应方案设计的第四阶段：详细描述关键业务流程。

本文围绕计划书要求，完整设计以下流程：

1. 系统初始化流程。
2. Connector 注册认证流程。
3. 数据发布流程。
4. 数据访问流程。
5. 属性更新流程。
6. 密钥生命周期管理流程。

每个流程都说明：

- 谁发起。
- 谁验证。
- 使用什么密钥。
- 返回什么结果。
- 哪些状态发生变化。
- Demo 前端应展示什么。
- 后端接口如何设计。

## 2. 流程参与方

| 参与方 | 简称 | 作用 |
| --- | --- | --- |
| 数据空间平台 | Platform | 编排业务流程，管理 Connector、资源、策略和访问请求。 |
| 连接器 | Connector | 模拟数据提供方和数据使用方。 |
| CA 认证机构 | CA | 签发和验证 Connector 证书。 |
| 属性授权机构 | AA | 管理属性、生成 ABE 属性密钥。 |
| 密钥管理服务 | KMS | 管理主密钥、DEK、密钥状态。 |
| 密码引擎 | Crypto Engine | 执行 AES、CP-ABE 加解密。 |
| 存储层 | Storage | 保存 Demo 状态、证书、密钥、资源和日志。 |
| 前端页面 | Frontend | 展示流程状态并触发操作。 |

## 3. 流程一：系统初始化流程

### 3.1 目标

建立 Demo 运行所需的根信任和初始状态：

- CA 初始化。
- AA 初始化。
- KMS 初始化。
- Platform 初始化。
- 初始属性定义初始化。

### 3.2 触发方式

前端系统总览页面点击：

```text
初始化系统
```

后端接口：

```http
POST /api/system/init
```

### 3.3 前置条件

```text
系统未初始化
或用户显式选择重新初始化
```

如果系统已初始化，接口应返回当前状态，或提示需要先 reset。

### 3.4 详细步骤

| 步骤 | 发起方 | 执行方 | 使用密钥 | 操作 | 结果 |
| --- | --- | --- | --- | --- | --- |
| 1 | Frontend | Platform | 无 | 发送初始化请求 | 请求进入后端 |
| 2 | Platform | CA | CA root key | 生成 CA 根密钥和根证书 | CA 状态为 `READY` |
| 3 | Platform | AA | ABE master key | 生成 ABE public key 和 master secret key | AA 状态为 `READY` |
| 4 | Platform | KMS | KMS master key | 生成 KMS master key | KMS 状态为 `READY` |
| 5 | Platform | Platform | 无 | 初始化资源目录、策略目录、主体目录 | Platform 状态为 `READY` |
| 6 | Platform | AA | 无 | 写入默认属性定义 | 属性定义可用 |
| 7 | Platform | Storage | 无 | 保存初始化状态 | Demo 状态持久化 |

### 3.5 默认属性定义

第一版 Demo 初始化以下属性：

```text
department=rd
department=sales
department=finance
role=researcher
role=auditor
role=admin
level=1
level=2
level=3
project=alpha
project=beta
```

### 3.6 返回结果

```json
{
  "system_status": "READY",
  "platform_status": "READY",
  "ca_status": "READY",
  "aa_status": "READY",
  "kms_status": "READY",
  "default_attributes": [
    "department=rd",
    "department=sales",
    "role=researcher",
    "role=auditor",
    "level=3"
  ]
}
```

### 3.7 前端展示

系统总览页面展示：

- Platform：READY
- CA：READY
- AA：READY
- KMS：READY
- Connector 数量：0
- 数据资源数量：0
- 密钥数量：至少 3 类系统密钥

## 4. 流程二：Connector 注册认证流程

### 4.1 目标

模拟 Connector 加入可信数据空间，建立身份可信关系。

### 4.2 触发方式

前端 Connector 管理页面点击：

```text
新增 Connector
```

后端接口：

```http
POST /api/connectors/register
```

### 4.3 请求示例

```json
{
  "name": "Connector A",
  "role": "PROVIDER",
  "attributes": ["department=rd", "role=researcher"]
}
```

### 4.4 前置条件

```text
系统已初始化
CA 状态 READY
AA 状态 READY
Platform 状态 READY
```

### 4.5 详细步骤

| 步骤 | 谁发起 | 谁验证 | 使用密钥 | 操作 | 返回结果 |
| --- | --- | --- | --- | --- | --- |
| 1 | Frontend/Connector | Platform | 无 | 提交 Connector 基本信息 | 注册请求 |
| 2 | Platform | Connector 模拟器 | Connector identity key | 生成 Connector 密钥对 | public_key/private_key_ref |
| 3 | Platform | CA | Connector public key, CA private key | 生成证书申请并签发证书 | certificate |
| 4 | Platform | CA | CA public key | 验证证书有效性 | certificate_status=VALID |
| 5 | Platform | Platform | 无 | 创建 Connector 记录 | connector_id |
| 6 | Platform | AA | AA 内部信任 | 绑定初始属性 | attribute_bindings |
| 7 | AA | AA | ABE master key | 生成初始 ABE 用户属性密钥 | abe_user_key |
| 8 | Platform | Storage | 无 | 保存 Connector、证书、属性、密钥记录 | 注册完成 |

### 4.6 状态变化

Connector 状态：

```text
PENDING -> CERTIFIED -> REGISTERED
```

证书状态：

```text
VALID
```

属性绑定状态：

```text
ACTIVE
```

ABE 用户密钥状态：

```text
ACTIVE
```

### 4.7 返回结果

```json
{
  "connector_id": "conn-a",
  "name": "Connector A",
  "role": "PROVIDER",
  "status": "REGISTERED",
  "certificate": {
    "certificate_id": "cert-001",
    "status": "VALID",
    "fingerprint": "sha256:..."
  },
  "attributes": ["department=rd", "role=researcher"],
  "abe_user_key_id": "key-abe-user-001",
  "steps": [
    "IDENTITY_KEY_GENERATED",
    "CERTIFICATE_ISSUED",
    "CERTIFICATE_VERIFIED",
    "CONNECTOR_REGISTERED",
    "ATTRIBUTES_BOUND",
    "ABE_USER_KEY_ISSUED"
  ]
}
```

### 4.8 异常分支

| 场景 | 返回原因 |
| --- | --- |
| 系统未初始化 | `SYSTEM_NOT_INITIALIZED` |
| CA 未就绪 | `CA_NOT_READY` |
| Connector 名称重复 | `CONNECTOR_ALREADY_EXISTS` |
| 属性不存在 | `ATTRIBUTE_NOT_DEFINED` |
| 证书签发失败 | `CERTIFICATE_ISSUE_FAILED` |

## 5. 流程三：数据发布流程

### 5.1 目标

模拟数据提供方 Connector A 发布数据，并完成：

- 生成 DEK。
- AES 加密数据。
- CP-ABE 加密 DEK。
- 保存密文资源。

### 5.2 触发方式

前端数据加密页面点击：

```text
加密并发布
```

后端接口：

```http
POST /api/data/encrypt
```

### 5.3 请求示例

```json
{
  "provider_connector_id": "conn-a",
  "name": "研发数据样例",
  "plaintext": "这是一份需要安全共享的数据",
  "abe_policy": "department=rd AND role=researcher"
}
```

### 5.4 前置条件

```text
Connector A 已注册
Connector A 证书有效
访问策略语法合法
KMS 可生成 DEK
AA 可提供 ABE public key
```

### 5.5 详细步骤

| 步骤 | 发起方 | 验证方/执行方 | 使用密钥 | 操作 | 结果 |
| --- | --- | --- | --- | --- | --- |
| 1 | Connector A | Platform | 无 | 提交明文和策略 | 发布请求 |
| 2 | Platform | CA | CA public key | 验证 Connector A 证书 | 身份可信 |
| 3 | Platform | Platform | 无 | 验证 Connector A 状态 | 可发布 |
| 4 | Platform | Policy Engine | 无 | 校验策略语法 | policy_valid=true |
| 5 | Platform | KMS | KMS master key | 生成 DEK | dek_key_id |
| 6 | Platform | Crypto Engine | DEK | AES-GCM 加密明文 | ciphertext |
| 7 | Platform | AA | ABE public key | 获取 ABE public key | abe_public_key |
| 8 | Platform | Crypto Engine | ABE public key | CP-ABE 加密 DEK | encrypted_dek |
| 9 | Platform | KMS | 无 | 保存 DEK 元数据 | key status ACTIVE |
| 10 | Platform | Storage | 无 | 保存资源记录 | resource status PUBLISHED |
| 11 | Platform | Audit Log | 无 | 记录发布日志 | log created |

### 5.6 资源状态变化

```text
DRAFT -> PUBLISHED
```

DEK 状态：

```text
ACTIVE
```

### 5.7 返回结果

```json
{
  "resource_id": "resource-001",
  "resource_status": "PUBLISHED",
  "dek_key_id": "key-dek-001",
  "key_version": 1,
  "encryption_algorithm": "AES-256-GCM",
  "abe_algorithm": "CP-ABE-DEMO",
  "abe_policy": "department=rd AND role=researcher",
  "ciphertext_preview": "u2FsdGVk...",
  "encrypted_dek_preview": "eyJw...",
  "steps": [
    "PROVIDER_CERT_VERIFIED",
    "PROVIDER_STATUS_CHECKED",
    "POLICY_VALIDATED",
    "DEK_GENERATED",
    "DATA_AES_ENCRYPTED",
    "DEK_CPABE_ENCRYPTED",
    "RESOURCE_PUBLISHED"
  ]
}
```

### 5.8 前端展示

数据加密页面展示流程条：

```text
生成 DEK -> AES 加密数据 -> CP-ABE 加密 DEK -> 资源发布完成
```

同时展示：

- 资源 ID
- DEK ID
- 策略
- 密文预览
- 加密 DEK 预览

### 5.9 异常分支

| 场景 | 返回原因 |
| --- | --- |
| Provider 不存在 | `PROVIDER_NOT_FOUND` |
| Provider 未注册 | `PROVIDER_NOT_REGISTERED` |
| 证书无效 | `CERTIFICATE_INVALID` |
| 策略语法错误 | `POLICY_INVALID` |
| KMS 生成 DEK 失败 | `DEK_GENERATION_FAILED` |
| AES 加密失败 | `AES_ENCRYPT_FAILED` |
| CP-ABE 加密失败 | `CPABE_ENCRYPT_FAILED` |

## 6. 流程四：数据访问流程

### 6.1 目标

模拟数据使用方 Connector B 请求访问资源，并展示：

- 身份认证。
- 权限检查。
- 属性匹配。
- ABE 解密 DEK。
- AES 解密数据。

### 6.2 触发方式

前端数据访问页面点击：

```text
请求访问
```

后端接口：

```http
POST /api/data/decrypt
```

### 6.3 请求示例

```json
{
  "consumer_connector_id": "conn-b",
  "resource_id": "resource-001"
}
```

### 6.4 前置条件

```text
Connector B 已注册
Connector B 证书有效
资源已发布
资源关联 DEK 状态 ACTIVE
Connector B 至少具有一组有效属性
```

### 6.5 详细步骤

| 步骤 | 发起方 | 验证方/执行方 | 使用密钥 | 操作 | 结果 |
| --- | --- | --- | --- | --- | --- |
| 1 | Connector B | Platform | 无 | 请求访问资源 | 访问请求 |
| 2 | Platform | CA | CA public key | 验证 Connector B 证书 | 证书有效 |
| 3 | Platform | Platform | 无 | 检查 Connector B 状态 | 主体有效 |
| 4 | Platform | Storage | 无 | 查询资源 | resource found |
| 5 | Platform | KMS | 无 | 检查 DEK 状态 | key ACTIVE |
| 6 | Platform | AA | 无 | 查询 Connector B 属性 | attributes |
| 7 | Platform | Policy Engine | 无 | 判断属性是否满足策略 | true/false |
| 8 | Platform | AA | ABE master key | 获取或生成 ABE 用户密钥 | abe_user_key |
| 9 | Platform | Crypto Engine | ABE user key | 解密 encrypted_dek | DEK |
| 10 | Platform | Crypto Engine | DEK | AES-GCM 解密数据 | plaintext |
| 11 | Platform | Audit Log | 无 | 记录访问结果 | access_log |

### 6.6 成功返回

```json
{
  "result": "SUCCESS",
  "plaintext": "这是一份需要安全共享的数据",
  "resource_id": "resource-001",
  "consumer_connector_id": "conn-b",
  "matched_policy": "department=rd AND role=researcher",
  "consumer_attributes": ["department=rd", "role=researcher"],
  "steps": [
    "CONSUMER_CERT_VERIFIED",
    "CONSUMER_STATUS_CHECKED",
    "RESOURCE_FOUND",
    "DEK_STATUS_ACTIVE",
    "ATTRIBUTES_LOADED",
    "POLICY_SATISFIED",
    "ABE_USER_KEY_READY",
    "DEK_CPABE_DECRYPTED",
    "DATA_AES_DECRYPTED"
  ]
}
```

### 6.7 失败返回：属性不满足

```json
{
  "result": "DENIED",
  "reason": "POLICY_NOT_SATISFIED",
  "resource_id": "resource-001",
  "required_policy": "department=rd AND role=researcher",
  "consumer_attributes": ["department=sales", "role=researcher"],
  "steps": [
    "CONSUMER_CERT_VERIFIED",
    "CONSUMER_STATUS_CHECKED",
    "RESOURCE_FOUND",
    "ATTRIBUTES_LOADED",
    "POLICY_NOT_SATISFIED"
  ]
}
```

### 6.8 失败原因表

| 失败点 | 原因码 |
| --- | --- |
| Connector 不存在 | `CONSUMER_NOT_FOUND` |
| Connector 未注册 | `CONSUMER_NOT_REGISTERED` |
| Connector 被禁用 | `CONSUMER_DISABLED` |
| 证书撤销 | `CERTIFICATE_REVOKED` |
| 资源不存在 | `RESOURCE_NOT_FOUND` |
| 资源被撤销 | `RESOURCE_REVOKED` |
| DEK 被撤销 | `DEK_REVOKED` |
| 属性不满足 | `POLICY_NOT_SATISFIED` |
| ABE 解密失败 | `CPABE_DECRYPT_FAILED` |
| AES 解密失败 | `AES_DECRYPT_FAILED` |

## 7. 流程五：属性更新流程

### 7.1 目标

模拟 Connector 属性变化，并展示权限变化。

典型场景：

```text
Connector B 原属性:
department=rd, role=researcher

变更为:
department=sales, role=researcher
```

原本满足策略：

```text
department=rd AND role=researcher
```

变更后不再满足。

### 7.2 触发方式

前端 Connector 管理页面编辑属性。

后端接口：

```http
PUT /api/connectors/{connector_id}/attributes
```

或：

```http
PUT /api/attributes/connectors/{connector_id}
```

### 7.3 请求示例

```json
{
  "attributes": ["department=sales", "role=researcher"],
  "reissue_abe_key": true
}
```

### 7.4 前置条件

```text
Connector 存在
Connector 状态 REGISTERED
AA 状态 READY
新属性都在属性定义表中
```

### 7.5 详细步骤

| 步骤 | 发起方 | 执行方 | 使用密钥 | 操作 | 结果 |
| --- | --- | --- | --- | --- | --- |
| 1 | 用户/前端 | Platform | 无 | 提交新属性集合 | 更新请求 |
| 2 | Platform | AA | 无 | 校验属性定义是否存在 | attributes valid |
| 3 | AA | Storage | 无 | 将旧属性绑定标记为 REVOKED | old bindings revoked |
| 4 | AA | Storage | 无 | 创建新属性绑定 | new bindings active |
| 5 | AA | KMS/Storage | 无 | 将旧 ABE_USER_SECRET_KEY 标记为 REVOKED | old key revoked |
| 6 | AA | Crypto Engine | ABE master key | 为新属性集合生成 ABE_USER_SECRET_KEY | new key active |
| 7 | Platform | Policy Engine | 无 | 重新计算 Connector 对资源的可访问性 | access preview |
| 8 | Platform | Audit Log | 无 | 记录属性变更日志 | log created |

### 7.6 状态变化

属性绑定：

```text
department=rd: ACTIVE -> REVOKED
department=sales: none -> ACTIVE
role=researcher: ACTIVE -> ACTIVE
```

ABE 用户密钥：

```text
key-abe-user-001: ACTIVE -> REVOKED
key-abe-user-002: none -> ACTIVE
```

### 7.7 返回结果

```json
{
  "connector_id": "conn-b",
  "old_attributes": ["department=rd", "role=researcher"],
  "new_attributes": ["department=sales", "role=researcher"],
  "revoked_abe_key_id": "key-abe-user-001",
  "new_abe_key_id": "key-abe-user-002",
  "access_preview": [
    {
      "resource_id": "resource-001",
      "policy": "department=rd AND role=researcher",
      "before": "ALLOWED",
      "after": "DENIED"
    }
  ],
  "steps": [
    "OLD_ATTRIBUTES_REVOKED",
    "NEW_ATTRIBUTES_BOUND",
    "OLD_ABE_KEY_REVOKED",
    "NEW_ABE_KEY_ISSUED",
    "ACCESS_RECALCULATED"
  ]
}
```

### 7.8 前端展示

Connector 管理页面展示：

- 旧属性。
- 新属性。
- 旧 ABE 密钥状态变为 REVOKED。
- 新 ABE 密钥状态为 ACTIVE。
- 访问权限变化。

数据访问页面可再次点击请求访问，展示失败：

```text
POLICY_NOT_SATISFIED
```

## 8. 流程六：密钥生命周期管理流程

### 8.1 目标

展示密钥从创建、分发、使用、更新、撤销到销毁的完整状态变化。

### 8.2 生命周期对象

第一版 Demo 重点管理三类密钥：

| 密钥 | 生命周期展示重点 |
| --- | --- |
| KMS master key | 初始化、轮换、状态展示。 |
| DEK | 生成、使用、轮换、撤销、销毁。 |
| ABE user key | 生成、属性变化后撤销和重发。 |

### 8.3 DEK 创建流程

触发点：

```text
数据发布流程
```

状态变化：

```text
none -> ACTIVE
```

记录：

```json
{
  "key_id": "key-dek-001",
  "key_type": "DATA_ENCRYPTION_KEY",
  "status": "ACTIVE",
  "version": 1
}
```

### 8.4 DEK 使用流程

DEK 使用发生在：

- AES 加密数据。
- AES 解密数据。

安全约束：

```text
明文 DEK 只在运行时短暂存在。
存储中不保存明文 DEK。
访问前必须检查 key status。
```

### 8.5 DEK 轮换流程

触发接口：

```http
POST /api/data/resources/{resource_id}/rekey
```

详细步骤：

| 步骤 | 执行方 | 操作 |
| --- | --- | --- |
| 1 | Platform | 查询资源和旧 DEK |
| 2 | KMS | 生成新 DEK，version + 1 |
| 3 | Crypto Engine | 使用旧流程恢复或读取明文数据 |
| 4 | Crypto Engine | 使用新 DEK AES-GCM 重新加密 |
| 5 | Crypto Engine | 使用 CP-ABE 按原策略加密新 DEK |
| 6 | KMS | 旧 DEK 状态改为 ROTATED |
| 7 | Platform | 资源 key_version 更新 |

返回示例：

```json
{
  "resource_id": "resource-001",
  "old_dek_key_id": "key-dek-001",
  "new_dek_key_id": "key-dek-002",
  "old_version": 1,
  "new_version": 2,
  "steps": [
    "NEW_DEK_GENERATED",
    "DATA_REENCRYPTED",
    "NEW_DEK_CPABE_ENCRYPTED",
    "OLD_DEK_ROTATED",
    "RESOURCE_KEY_VERSION_UPDATED"
  ]
}
```

### 8.6 DEK 撤销流程

触发接口：

```http
POST /api/keys/{key_id}/revoke
```

状态变化：

```text
ACTIVE -> REVOKED
```

影响：

```text
关联资源不能继续解密。
数据访问流程在 KMS 检查阶段失败。
```

访问失败返回：

```json
{
  "result": "DENIED",
  "reason": "DEK_REVOKED"
}
```

### 8.7 DEK 销毁流程

触发接口：

```http
POST /api/keys/{key_id}/destroy
```

状态变化：

```text
REVOKED -> DESTROYED
```

Demo 处理：

- 不真实删除审计元数据。
- `material_ref` 标记为 `destroyed`。
- 资源显示不可恢复。

### 8.8 ABE 用户密钥撤销流程

触发点：

```text
Connector 属性更新
Connector 撤销
证书撤销
```

状态变化：

```text
ACTIVE -> REVOKED
```

访问影响：

```text
即使缓存中存在旧 ABE key，也不能用于访问。
```

### 8.9 KMS master key 轮换流程

Demo 中可展示但不强制影响所有密钥材料。

触发接口：

```http
POST /api/keys/{master_key_id}/rotate
```

状态变化：

```text
key-kms-master-001: ACTIVE -> ROTATED
key-kms-master-002: none -> ACTIVE
```

影响：

- 新 DEK 关联新 master key。
- 旧 DEK 保留原 parent key。
- 前端展示主密钥版本变化。

## 10. Demo 演示脚本

### 10.1 成功访问场景

```text

3. 注册 Connector B:

   policy=department=rd AND role=researcher

### 10.2 属性变化导致访问失败

   department=sales, role=researcher

3. 展示属性不满足策略。
4. 展示旧 ABE key REVOKED，新 ABE key ACTIVE。
```

### 10.3 密钥撤销导致访问失败

```text
1. 恢复 Connector B 属性:
   department=rd, role=researcher

3. Connector B 再次请求访问。

```

### 10.4 资源重加密场景

```text
1. 对 resource-001 执行 rekey。
2. 生成 key-dek-002。
3. 资源 key_version 从 1 变为 2。

```

## 12. 第四阶段结论

本阶段已经把 Demo 的关键业务流程细化为可实现方案。

后续编码时，应优先保证以下最短闭环：

```text
初始化系统
  -> 注册 Connector A / B
  -> Connector A 发布加密数据
  -> Connector B 属性满足时访问成功
  -> 修改 Connector B 属性
  -> Connector B 再次访问失败
  -> 撤销/轮换 DEK
  -> 密钥管理页面可观察状态变化
```

这个闭环正好对应计划书的核心目标：

```text
完整展示可信数据空间中，基于 CP-ABE 和密钥管理的数据安全共享流程。
```
