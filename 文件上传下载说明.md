# 文件上传下载说明

## 1. 当前文件流程

当前系统采用“Connector 私有目录 + 系统密文目录”的两层文件模型。

第一层是 Connector 自己的本地文件目录：

```text
storage/connectors/{connectorId}
```

它用于模拟某个 Connector 本地拥有的文件。每个 Connector 注册成功后，系统都会为它创建一个独立目录，初始为空。

第二层是系统保存加密资源的目录：

```text
storage/files
```

它用于保存已经发布到系统中的加密文件密文包。

## 2. 正确的上传流程

现在文件上传分两步。

第一步：本地文件导入 Connector 私有目录。

```text
本机文件
  -> 选择 Connector
  -> 导入到 storage/connectors/{connectorId}
  -> 数据库记录 connector_files
```

这一步只是模拟 Connector 拥有了这个文件，还没有发布给系统。

第二步：Provider 从自己的目录中选择文件，发布到系统。

```text
Connector 私有目录文件
  -> 选择访问策略
  -> 生成 DEK
  -> AES-GCM 加密文件内容
  -> CP-ABE Demo 保护 DEK
  -> 加密文件包保存到 storage/files
  -> 系统生成文件资源
```

## 3. 下载流程

Consumer 下载系统文件时，流程如下：

```text
选择 Consumer
  -> 选择系统文件资源
  -> 检查 Consumer 证书和状态
  -> 检查资源 DEK 是否 ACTIVE
  -> 检查 Consumer 属性是否满足访问策略
  -> 满足则恢复 DEK
  -> 解密 storage/files 中的密文文件包
  -> 文件写入 Consumer 私有目录
  -> 浏览器触发下载
```

如果属性不满足，会返回：

```text
POLICY_NOT_SATISFIED
```

如果 DEK 被撤销，会返回：

```text
DEK_REVOKED
```

## 4. 推荐测试流程

1. 点击“重新初始化”。
2. 手动新增 Provider，例如：

```text
name=Connector A
role=PROVIDER
attributes=department=rd,role=researcher,level=3
```

3. 手动新增 Consumer，例如：

```text
name=Connector B
role=CONSUMER
attributes=department=rd,role=researcher
```

4. 在“Connector 本地文件目录”中选择 Connector A。
5. 选择本机文件，点击“导入到目录”。
6. 在“从 Connector 发布文件到系统”中选择 Connector A 和它目录中的文件。
7. 设置访问策略：

```text
department=rd AND role=researcher
```

8. 点击“发布文件”。
9. 在“系统文件下载”中选择 Connector B 和系统文件资源。
10. 点击“按规则下载文件”。

此时应该下载成功，并且系统会把解密后的文件也写入 Connector B 的私有目录。

## 5. 验证属性控制

点击：

```text
将 Consumer 改为销售属性
```

再下载同一个系统文件。

如果资源策略仍然是：

```text
department=rd AND role=researcher
```

则下载应该失败：

```text
POLICY_NOT_SATISFIED
```

这说明属性控制已经影响了 DEK 恢复过程。

## 6. 验证密钥控制

点击：

```text
恢复研发属性
```

然后选择同一个系统文件，点击：

```text
撤销当前资源 DEK
```

再下载该文件。

此时即使 Consumer 属性满足策略，也应该失败：

```text
DEK_REVOKED
```

这说明密钥状态也会影响文件能否解密。

## 7. 数据库存储了什么

数据库不会保存完整文件内容。

数据库只保存文件元数据，例如：

```text
connector_file_id
connector_id
file_name
mime_type
file_size
local_path
origin
status
published_resource_id
```

真正的文件内容保存在本地目录中。

## 8. 文件大小限制

当前前端使用 JSON + Base64 传输文件，适合测试小文件。

默认请求体限制是：

```text
25MB
```

可以在 `.env` 中修改：

```text
MAX_JSON_BODY_BYTES=26214400
```

如果后续要上传大文件，建议改成 multipart 上传，或者接入 MinIO。

## 9. 后续替换 MinIO 的思路

当前本地存储模块分为：

```text
backend/storage/connectorFileStorage.js
backend/storage/localFileStorage.js
```

以后可以增加：

```text
backend/storage/minioStorage.js
```

把 `storage/files` 中的系统密文文件改为存入 MinIO。Connector 私有目录可以继续本地模拟，也可以进一步抽象为不同 Connector 的对象存储 Bucket。
