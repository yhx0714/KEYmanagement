# 文件上传下载说明

## 1. 当前实现方式

项目现在已经支持使用本机文件进行上传和下载测试。

当前采用的是“本地目录模拟对象存储”的方式：

```text
上传文件
  -> 后端读取文件内容
  -> 生成 DEK
  -> 使用 AES-GCM 加密文件内容
  -> 使用访问策略保护 DEK
  -> 将加密后的文件密文包保存到本地目录
```

本地密文目录是：

```text
storage/files
```

注意：这里保存的是加密后的密文文件包，不是原始明文文件。

## 2. 上传文件时发生什么

在页面中选择本地文件，然后点击：

```text
上传并加密
```

系统会执行：

1. 前端读取本地文件内容。
2. 前端把文件内容转成 Base64 后提交给后端。
3. 后端验证 Provider Connector。
4. 后端校验访问策略。
5. KMS 生成一个新的 DEK。
6. 后端使用 AES-GCM 和 DEK 加密文件内容。
7. 后端使用 CP-ABE Demo 策略保护 DEK。
8. 后端将加密后的文件密文包保存到 `storage/files`。
9. 系统生成一个文件资源记录。

## 3. 下载文件时发生什么

在页面中选择一个文件资源，然后点击：

```text
下载文件
```

系统会执行：

1. 验证 Consumer Connector 是否有效。
2. 查询文件资源。
3. 检查资源对应的 DEK 是否仍然是 `ACTIVE`。
4. 读取 Consumer 的属性。
5. 判断属性是否满足资源访问策略。
6. 如果满足策略，恢复 DEK。
7. 从 `storage/files` 读取文件密文包。
8. 使用 DEK 解密文件内容。
9. 返回文件内容给浏览器。
10. 浏览器触发本地下载。

如果属性不满足，会返回：

```text
POLICY_NOT_SATISFIED
```

如果 DEK 被撤销，会返回：

```text
DEK_REVOKED
```

## 4. 测试建议

推荐按这个流程测试：

1. 启动项目。
2. 点击“生成演示数据”。
3. 在“文件上传加密”区域选择一个本地小文件。
4. 使用默认策略：

```text
department=rd AND role=researcher
```

5. 点击“上传并加密”。
6. 在资源下拉框中选择 `[文件]` 开头的资源。
7. 点击“下载文件”。
8. 检查下载后的文件内容是否和原文件一致。

然后继续测试属性控制：

1. 点击“将 Consumer 改为销售属性”。
2. 再次选择同一个文件资源。
3. 点击“下载文件”。
4. 观察结果应为 `POLICY_NOT_SATISFIED`。

再测试密钥控制：

1. 点击“恢复研发属性”。
2. 选择文件资源。
3. 点击“撤销当前资源 DEK”。
4. 再次点击“下载文件”。
5. 观察结果应为 `DEK_REVOKED`。

## 5. 文件大小限制

当前前端用 JSON + Base64 上传文件，因此适合测试小文件。

默认请求体限制是：

```text
25MB
```

可以在 `.env` 中修改：

```text
MAX_JSON_BODY_BYTES=26214400
```

如果后续要上传大文件，建议改成真正的 multipart 文件上传，或者直接接入 MinIO。

## 6. 后续如何替换成 MinIO

当前本地文件存储模块在：

```text
backend/storage/localFileStorage.js
```

它提供两个核心动作：

```text
writeEncryptedFile
readEncryptedFile
```

后续如果要换成 MinIO，可以新增：

```text
backend/storage/minioStorage.js
```

保持类似接口：

```text
writeEncryptedFile(resourceId, encryptedData)
readEncryptedFile(storagePath)
```

业务层不用大改，只需要把本地存储模块替换成 MinIO 存储模块。

## 7. 当前实现的重点

当前实现已经可以说明：

```text
文件内容不是明文保存
文件上传后先被 AES-GCM 加密
DEK 受访问策略保护
下载文件前必须满足属性策略
DEK 被撤销后无法下载解密
```

也就是说，文件上传/下载流程已经纳入了同一套密钥管理和属性控制逻辑。
