# 可信数据空间密钥管理与加密演示系统：技术调研与方案分析

## 1. 文档目标

本文档对应任务的第一部分：技术调研与方案分析。

本 Demo 的目标不是直接构建生产级可信数据空间，而是验证以下核心思想：

1. 数据在可信数据空间中如何加密存储和传输。
2. 连接器之间如何通过证书和平台注册建立基本信任。
3. 平台、CA、属性授权机构、KMS 如何分工。
4. 如何使用 `CP-ABE + 对称加密` 实现细粒度访问控制。
5. 如何设计一个后续可扩展到真实连接器和真实平台的密钥管理模型。

因此，本阶段重点不是写代码，而是先形成技术依据和方案边界，为后续架构设计、接口设计和 Demo 开发提供依据。

## 2. 已有基础分析

当前 `生产实习` 目录下已有两个可参考项目或文档：

## 3. 可信数据空间安全模型调研

可信数据空间通常强调“数据不离控制、主体可信、策略可执行、过程可追溯”。在加密与密钥管理层面，它至少需要解决四类问题：

1. **身份可信**：谁是平台、谁是连接器、谁是用户。
2. **通信可信**：连接器与平台、连接器与连接器之间是否能建立安全通道。
3. **访问可信**：请求方是否满足访问策略。
4. **密钥可信**：数据密钥、属性密钥、证书私钥、主密钥是否被正确生成、分发、更新、撤销和销毁。

国际数据空间、去中心化身份和企业级 KMS 的设计都给了可参考的方向：

- 数据空间中的 Connector 应当被平台登记、识别和治理。
- 主体身份可使用 PKI 证书、DID 或平台账号模型表达。
- 数据访问不能只依赖“是否登录”，还需要结合属性、策略、合同、用途等上下文。
- 数据本体应使用高效对称加密，访问权限则由密钥封装或策略控制实现。
- 密钥管理应覆盖生成、存储、使用、轮换、撤销、归档和销毁。

本 Demo 选择较清晰、易展示的集中式模型：

```text
CA 负责身份可信
Attribute Authority 负责属性可信
KMS 负责密钥生命周期
Platform 负责主体注册、资源目录和策略管理
Connector 负责模拟数据发布与访问
Crypto Engine 负责 AES 与 CP-ABE 加解密
```

## 4. KMS 技术调研

KMS，即 Key Management Service，核心职责是管理密码材料的全生命周期。

参考 NIST SP 800-57 Part 1，密钥管理应覆盖密钥从生成到销毁的生命周期，密钥安全直接影响加密机制整体安全。参考 NIST SP 800-130，CKMS 设计应明确密钥类型、元数据、访问控制、生命周期状态、审计、恢复、销毁等主题。

### 4.1 KMS 中常见密钥类型

| 密钥类型 | 说明 | Demo 中的对应设计 |
| --- | --- | --- |
| Root Key / Master Key | KMS 内部最高级别主密钥，用于保护下级密钥。 | `kms_master_key`，只在 KMS 内部使用。 |
| Key Encryption Key, KEK | 用于包裹或加密其他密钥。 | 可由 master key 派生或直接模拟。 |
| Data Encryption Key, DEK | 用于加密具体数据。 | 每次数据发布随机生成 AES DEK。 |
| ABE Public Key | ABE 系统公开参数。 | 由 Attribute Authority 初始化生成。 |
| ABE Master Secret Key | ABE 主密钥，用于生成属性私钥。 | 仅由 Attribute Authority 持有。 |
| ABE User Secret Key | 与用户/连接器属性绑定的私钥。 | 按 Connector 属性生成，可缓存并带生命周期状态。 |
| Connector Identity Private Key | 连接器身份私钥。 | 由 Connector 模拟生成，用于申请证书和签名。 |
| Certificate Public Key | 连接器证书中的公钥。 | 由 CA 签发证书后公开。 |

### 4.2 KMS 应管理的密钥元数据

Demo 中每个密钥应至少包含：

| 字段 | 说明 |
| --- | --- |
| `key_id` | 密钥唯一标识。 |
| `key_type` | 密钥类型，如 `MASTER`、`DEK`、`ABE_USER_KEY`。 |
| `owner_id` | 所属主体，如 Platform、Connector A。 |
| `algorithm` | 算法，如 `AES-256-GCM`、`CP-ABE-BSW`。 |
| `status` | 状态，如 `ACTIVE`、`DISABLED`、`REVOKED`、`DESTROYED`。 |
| `created_at` | 创建时间。 |
| `expires_at` | 过期时间，可选。 |
| `version` | 密钥版本。 |
| `usage` | 用途，如 `ENCRYPT`、`DECRYPT`、`WRAP`、`SIGN`。 |
| `policy_id` | 关联策略，可选。 |
| `revoked_reason` | 撤销原因，可选。 |

### 4.3 密钥生命周期

Demo 应模拟以下生命周期状态：

```text
PRE_ACTIVE -> ACTIVE -> SUSPENDED/DISABLED -> REVOKED -> DESTROYED
```

更适合 Demo 的简化版本：

| 状态 | 含义 |
| --- | --- |
| `ACTIVE` | 可用于加密/解密。 |
| `DISABLED` | 暂停使用，但可恢复。 |
| `REVOKED` | 已撤销，不能再用于新的授权或解密。 |
| `ROTATED` | 已轮换，保留旧版本用于历史数据。 |
| `DESTROYED` | 已销毁，不能恢复。 |

### 4.4 Demo 中 KMS 的边界

生产级 KMS 通常需要硬件安全模块、访问控制策略、密钥包装、密钥审计和灾备。本 Demo 不实现这些复杂能力，但要保留接口和状态模型。

本 Demo 中 KMS 负责：

- 生成 AES DEK。
- 记录 DEK 元数据。
- 保存 DEK 的密文封装结果，而不是长期保存明文 DEK。
- 管理 DEK 生命周期。
- 管理 ABE 用户密钥记录。
- 模拟密钥轮换、撤销和销毁。

KMS 不直接负责：

- 真实 HSM 保护。
- 真实云 KMS 对接。
- 真实密钥恢复审批。
- 生产级多租户隔离。

## 5. PKI 与 CA 技术调研

PKI 负责主体身份可信。它通常包括：

| 组件 | 作用 |
| --- | --- |
| Root CA | 信任锚，签发下级 CA 或直接签发证书。 |
| Intermediate CA | 中间 CA，降低 Root CA 暴露风险。 |
| Registration Authority | 注册审核，可检查主体信息。 |
| Certificate Repository | 证书发布与查询。 |
| CRL / OCSP | 证书撤销状态查询。 |

在可信数据空间中，CA 可用于：

- 给 Platform 签发证书。
- 给 Connector 签发证书。
- 证明 Connector 的公钥与身份绑定。
- 支持连接器注册时的身份验证。
- 支持后续 mTLS 或签名认证扩展。

### 5.1 Demo 中的 CA 模型

为了保持轻量，Demo 中 CA 只做抽象模拟：

```text
CA 初始化
  -> 生成 CA 根密钥对
  -> 生成 Root CA 证书

Connector 注册
  -> Connector 生成自己的密钥对
  -> Connector 提交 CSR 风格申请
  -> CA 验证申请信息
  -> CA 签发 Connector 证书
  -> Platform 记录证书指纹和状态
```

### 5.2 证书状态

Demo 中证书至少应包含：

| 字段 | 说明 |
| --- | --- |
| `certificate_id` | 证书 ID。 |
| `subject_id` | 证书主体，如 Connector ID。 |
| `subject_name` | 主体名称。 |
| `public_key` | 主体公钥，Demo 可用字符串模拟。 |
| `issuer` | 签发者，如 Demo Root CA。 |
| `serial_number` | 序列号。 |
| `valid_from` | 生效时间。 |
| `valid_to` | 失效时间。 |
| `status` | `VALID`、`REVOKED`、`EXPIRED`。 |
| `fingerprint` | 证书摘要。 |

## 6. 属性授权机构与 ABE 调研

ABE，即 Attribute-Based Encryption，属性基加密，是一种通过属性和策略控制解密权限的公钥加密思想。

常见 ABE 分为：

| 类型 | 策略位置 | 特点 |
| --- | --- | --- |
| KP-ABE | 策略在用户私钥中，数据带属性。 | 更像授权方控制用户能访问什么类型数据。 |
| CP-ABE | 策略在密文中，用户私钥带属性。 | 更适合数据发布者定义访问规则。 |

本 Demo 选择 **CP-ABE**。

原因：

1. 数据提供方发布数据时可以直接定义访问策略。
2. 用户/连接器只需要持有属性私钥。
3. 适合展示“属性满足策略才能解密”的过程。
4. 与已有 `ABE加密服务` 的设计一致。

### 6.1 CP-ABE 基本算法

CP-ABE 抽象包含四个核心算法：

```text
Setup() -> public_key, master_secret_key
KeyGen(public_key, master_secret_key, attributes) -> user_secret_key
Encrypt(public_key, access_policy, message_or_key) -> ciphertext
Decrypt(public_key, user_secret_key, ciphertext) -> message_or_key
```

在本 Demo 中，不直接用 CP-ABE 加密大数据，而是使用混合加密：

```text
AES 加密数据
CP-ABE 加密 AES 数据密钥 DEK
```

### 6.2 属性授权机构职责

Attribute Authority 简称 AA，负责属性可信。

Demo 中 AA 的职责：

- 定义属性命名空间，如 `role=researcher`、`department=rd`、`level=3`。
- 给 Connector 绑定属性。
- 根据 Connector 当前属性生成 ABE 用户私钥。
- 属性变化后触发 ABE 私钥更新或撤销。
- 向 Platform/KMS 同步属性状态。

AA 与 KMS 的边界：

| 机构 | 管什么 |
| --- | --- |
| AA | ABE 系统主密钥、ABE 公钥、属性定义、属性私钥生成。 |
| KMS | AES DEK、密钥生命周期、密钥状态、密钥审计。 |

在真实系统中，AA 和 KMS 可以合并，也可以分离。Demo 中建议逻辑分离，便于展示职责。

## 7. 对称加密技术调研

大文件或业务数据一般不直接使用公钥加密，而使用对称加密。

### 7.1 AES

AES 是最常见的对称分组密码。对数据安全共享场景，推荐使用带认证的加密模式，例如 AES-GCM。

参考 NIST SP 800-38D，GCM 是一种认证加密模式，同时提供保密性与完整性校验。已有 `ABE加密服务` 对外 AES 接口使用的是 `AES/GCM/NoPadding`，因此 Demo 也建议沿用：

```text
AES-256-GCM
```

加密输出结构：

```text
nonce/iv + ciphertext + authentication tag
```

### 7.2 国密算法预留

计划书要求预留 SM2、SM3、SM4 接口。

Demo 中建议这样处理：

| 国密算法 | 作用 | Demo 状态 |
| --- | --- | --- |
| SM2 | 非对称加密/签名，可替代部分 RSA/ECC 身份认证能力。 | 预留接口，不强制实现。 |
| SM3 | 哈希摘要，可用于证书指纹、数据摘要。 | 可先用 SHA-256 模拟，保留 `hash_algorithm` 字段。 |
| SM4 | 对称加密，可替代 AES。 | 预留 `SM4-GCM` 或 `SM4-CBC` 算法标识。 |

为了最小可运行，第一版 Demo 实现 AES 即可，接口层保留算法枚举：

```text
AES-256-GCM
SM4-GCM-PLACEHOLDER
```

## 8. 数据传输加密流程调研

可信数据空间中的数据传输可以分为两层：

1. 通信信道加密：HTTPS/mTLS，保护传输通道。
2. 数据对象加密：数据本身被 AES/CP-ABE 保护，即使离开传输通道仍保持密文状态。

本 Demo 重点展示第二层，即对象级加密。

建议模型：

```text
Connector A 发布数据
  -> 本地生成 DEK
  -> AES-GCM 加密明文
  -> CP-ABE 加密 DEK
  -> Platform 保存资源元数据和访问策略
  -> Storage 保存密文数据和加密后的 DEK

Connector B 请求数据
  -> Platform 校验证书状态和主体状态
  -> AA 校验属性状态
  -> Connector B 获取或生成 ABE 属性私钥
  -> 解开加密 DEK
  -> AES-GCM 解密数据
```

通信信道加密在 Demo 中不实现真实 TLS，但文档和接口保留：

- `certificate_id`
- `certificate_status`
- `connector_public_key`
- `signature`
- `mTLS_ready`

## 9. 密钥生命周期管理方案调研

密钥生命周期是本 Demo 的核心展示点之一。

### 9.1 主密钥生命周期

主密钥用于保护 KMS 内部数据，生命周期长，但权限最高。

Demo 简化策略：

| 动作 | 说明 |
| --- | --- |
| 创建 | 系统初始化时生成。 |
| 存储 | Demo 内存或本地文件中保存，后续可替换为 Vault/HSM。 |
| 使用 | 只用于派生或保护下级密钥。 |
| 轮换 | 支持生成新版本 master key，旧 key 保留解历史数据。 |
| 撤销 | 标记旧版本不可用于新密钥保护。 |
| 销毁 | Demo 中仅改变状态，不真实删除，便于展示。 |

### 9.2 数据密钥 DEK 生命周期

DEK 是每份数据或每次发布生成的 AES 密钥。

```text
生成 -> 用于 AES 加密 -> 被 CP-ABE 包裹 -> 明文 DEK 从运行时清理 -> 加密 DEK 随资源保存
```

Demo 中 DEK 元数据状态：

| 状态 | 说明 |
| --- | --- |
| `ACTIVE` | 当前资源可正常解密。 |
| `ROTATED` | 资源已重新加密，新 DEK 生效。 |
| `REVOKED` | 不允许继续解密。 |
| `DESTROYED` | 模拟销毁。 |

### 9.3 ABE 属性私钥生命周期

ABE 用户私钥与属性集合绑定。

当 Connector 属性变化时：

```text
旧属性私钥 -> REVOKED
新属性集合 -> 重新 KeyGen -> 新属性私钥 ACTIVE
```

ABE 撤销是难点。常见方案包括：

- 短周期属性密钥。
- 策略中加入时间属性，例如 `date=2026-07`。
- 属性变更后重新加密 DEK。
- 维护撤销列表并在访问前由平台过滤。

Demo 建议采用“平台过滤 + 属性密钥重发 + 可选重新加密”的简化方案：

```text
属性变化后，旧 ABE key 标记为 REVOKED。
访问请求必须先经过 Platform/AA 检查。
如果资源需要强撤销，则重新生成 DEK 并重新加密数据。
```

## 10. 技术方案对比

### 10.1 纯 AES 方案

| 项 | 说明 |
| --- | --- |
| 优点 | 实现简单，性能高。 |
| 缺点 | 每个授权对象都要分发密钥，细粒度授权困难。 |
| 适用场景 | 小规模、固定授权关系。 |
| 是否适合 Demo | 不足以展示属性访问控制，只作为数据本体加密。 |

### 10.2 传统 PKI + AES 密钥封装

| 项 | 说明 |
| --- | --- |
| 优点 | 工程成熟，易与证书体系结合。 |
| 缺点 | 数据发布者需要面向每个接收方封装 DEK，授权变化复杂。 |
| 适用场景 | 明确接收方列表的数据共享。 |
| 是否适合 Demo | 适合作为身份层，不作为细粒度授权核心。 |

### 10.3 CP-ABE + AES 混合加密

| 项 | 说明 |
| --- | --- |
| 优点 | 策略写入密文，数据提供方可定义属性策略，适合细粒度访问控制。 |
| 缺点 | 撤销复杂，性能低于传统公钥体系，策略可能泄露。 |
| 适用场景 | 属性驱动的数据访问控制。 |
| 是否适合 Demo | 非常适合，能直观展示可信数据空间安全共享思想。 |

### 10.4 真实 KMS/HSM 方案

| 项 | 说明 |
| --- | --- |
| 优点 | 安全性高，符合生产实践。 |
| 缺点 | 部署复杂，不利于轻量 Demo。 |
| 适用场景 | 生产系统。 |
| 是否适合 Demo | 第一版不直接实现，只保留接口和模型。 |

## 11. 推荐 Demo 技术选型

结合计划书要求和当前项目基础，推荐第一版 Demo 采用：

| 层 | 选型 | 原因 |
| --- | --- | --- |
| 后端框架 | Python FastAPI | 结构清晰，接口文档自动生成，适合快速 Demo。 |
| 前端 | 简单 HTML + JS 或 Vite + React | 展示流程为主，避免复杂 UI。 |
| 加密实现 | Python `cryptography` AES-GCM + 模拟 CP-ABE 或复用 Java ABE 思路 | Demo 需要可运行，CP-ABE 可先做逻辑模拟，后续替换真实实现。 |
| 存储 | JSON 文件或 SQLite | 轻量、可观察、适合演示密钥状态。 |
| API 风格 | REST | 与计划书一致，便于前端调用。 |
| 部署 | 本地运行 | 不依赖真实连接器和平台。 |

### 11.1 关于 CP-ABE 实现的建议

真实 CP-ABE 库在 Python 生态中部署成本相对高，尤其是 pairing 库依赖。因此第一版 Demo 有两个可选路径：

| 路径 | 描述 | 优点 | 缺点 |
| --- | --- | --- | --- |
| A. 真实 CP-ABE | 调用已有 Java ABE 服务或嵌入 pairing 库。 | 更接近密码学真实过程。 | 集成复杂，环境依赖重。 |
| B. 演示型 CP-ABE | 实现策略解析、属性匹配和 DEK 包裹模拟。 | 快速稳定，可完整展示流程。 | 密码学上不是生产级 ABE。 |

建议本 Demo 第一版采用 **B：演示型 CP-ABE**，但接口设计保持真实 CP-ABE 形态：

```text
setup()
keygen(attributes)
encrypt_dek(dek, policy)
decrypt_dek(abe_ciphertext, user_key)
```

后续可以把内部实现替换为已有 `ABE加密服务` 或更成熟的 ABE 库，而不影响上层流程。

## 12. 推荐总体方案

最终推荐方案：

```text
身份层：模拟 CA + Connector 证书
属性层：Attribute Authority 管理属性与 ABE 用户密钥
密钥层：KMS 管理 master key、DEK、key version、key status
数据层：AES-256-GCM 加密数据
访问控制层：CP-ABE 保护 DEK，策略绑定到数据资源
平台层：Platform 管理 Connector、数据资源、策略和访问流程
展示层：前端页面展示每一步状态
```

核心加密流程：

```text
明文数据
  -> KMS 生成随机 DEK
  -> AES-GCM 使用 DEK 加密数据
  -> Attribute Authority 使用 CP-ABE policy 加密 DEK
  -> 保存 encrypted_data + encrypted_dek + policy + metadata
```

核心解密流程：

```text
Connector B 请求资源
  -> Platform 校验证书与连接器状态
  -> Attribute Authority 校验属性状态
  -> 生成或获取 Connector B 的 ABE 属性私钥
  -> CP-ABE 解密 encrypted_dek 得到 DEK
  -> AES-GCM 解密 encrypted_data 得到明文
```

## 13. 第一阶段结论

根据调研和当前任务目标，后续 Demo 应围绕以下原则展开：

1. **用 PKI/CA 表达主体可信**：Connector 必须先注册并获得证书。
2. **用 AA 表达属性可信**：属性不是用户自报，而是由属性授权机构管理。
3. **用 KMS 管理密钥生命周期**：所有密钥都要有类型、状态、版本和用途。
4. **用 AES 加密数据本体**：保证效率和清晰工程实现。
5. **用 CP-ABE 控制 DEK 解密权限**：展示属性驱动访问控制。
6. **用平台编排流程**：Platform 不直接持有所有明文密钥，但负责注册、资源、策略和访问流程。
7. **第一版优先演示完整流程**：不要过早陷入真实连接器、真实 HSM、复杂 UI。

```text
02-系统架构与形式化模型设计
```

该阶段将基于本文调研结论，正式定义实体、数据模型、模块边界、接口草案和关键业务流程。
