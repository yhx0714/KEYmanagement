# 可信数据空间密钥管理与加密演示系统：系统架构与形式化模型设计

## 1. 文档目标

本文档对应任务的第二阶段：系统架构与形式化模型设计。

在第一阶段技术调研的基础上，本文将 Demo 系统抽象为一组实体、状态、数据结构、接口和业务流程，为后续编码提供明确边界。

本文档解决以下问题：

1. Demo 系统包含哪些核心实体。
2. 各实体之间如何建立信任关系。
3. 密钥、证书、属性、策略和数据资源如何建模。
4. CP-ABE + AES 混合加密如何落入具体流程。
5. 后端 API 应如何规划。
6. 前端演示页面应展示哪些状态和步骤。

本阶段仍然不实现代码。

## 2. 总体架构

Demo 系统采用单机可运行的模拟架构，逻辑上拆分为六个核心模块：

```text
┌─────────────────────────────────────────────────────────────┐
│                         Frontend Demo                        │
│  Overview / Connector / Encrypt / Access / Key Management    │
└──────────────────────────────┬──────────────────────────────┘
                               │ REST API
┌──────────────────────────────▼──────────────────────────────┐
│                         API Gateway                          │
│                 FastAPI / Flask Controller Layer             │
└───────┬─────────────┬─────────────┬─────────────┬───────────┘
        │             │             │             │
┌───────▼──────┐ ┌────▼─────┐ ┌─────▼──────┐ ┌────▼──────────┐
│   Platform   │ │    CA    │ │ Attribute  │ │      KMS      │
│              │ │ Service  │ │ Authority  │ │               │
└───────┬──────┘ └────┬─────┘ └─────┬──────┘ └────┬──────────┘
        │             │             │             │
        │             │             │             │
┌───────▼─────────────▼─────────────▼─────────────▼───────────┐
│                         Crypto Engine                        │
│           AES-GCM / CP-ABE Demo Engine / SM Placeholder      │
└──────────────────────────────┬──────────────────────────────┘
                               │
┌──────────────────────────────▼──────────────────────────────┐
│                      Demo Storage Layer                      │
│        JSON / SQLite: connectors, keys, certs, data          │
└─────────────────────────────────────────────────────────────┘
```

### 2.1 模块职责

| 模块 | 职责 |
| --- | --- |
| Frontend Demo | 展示系统状态、连接器注册、属性变更、数据加密、数据访问、密钥生命周期。 |
| API Gateway | 统一暴露 REST API，对接前端。 |
| Platform | 管理 Connector 注册、资源目录、访问策略、访问请求编排。 |
| CA Service | 初始化 CA、签发 Connector 证书、管理证书状态。 |
| Attribute Authority | 管理属性定义、属性绑定、ABE 系统密钥和属性私钥。 |
| KMS | 管理主密钥、数据密钥、密钥状态、版本、撤销和销毁。 |
| Crypto Engine | 实现或模拟 AES、CP-ABE、国密算法接口。 |
| Demo Storage | 保存 Demo 运行状态，第一版建议 JSON 或 SQLite。 |

## 3. 设计原则

本 Demo 遵循以下原则：

1. **实体清晰**：Platform、CA、AA、KMS、Connector 职责分离。
2. **流程完整**：必须能展示初始化、注册、加密、访问、属性变更、密钥撤销。
3. **密码可替换**：第一版可使用演示型 CP-ABE，但接口保持真实 CP-ABE 形态。
4. **状态可观察**：所有密钥、证书、连接器、资源都有生命周期状态。
5. **不做真实连接器**：Connector 是软件实体和数据结构，不运行独立进程。
6. **不追求复杂 UI**：前端只服务于流程展示。
7. **面向扩展**：后续可替换为真实 ABE 服务、真实 CA、真实 KMS、真实连接器。

## 4. 核心实体定义

### 4.1 Platform

数据空间平台是 Demo 的业务编排中心。

职责：

- 管理可信主体。
- 管理 Connector 注册信息。
- 管理数据资源目录。
- 管理访问策略。
- 管理访问请求。
- 协调 CA、AA、KMS 和 Crypto Engine。

Platform 不直接长期保存明文 DEK，也不生成 ABE 属性私钥。

### 4.2 Connector

Connector 是可信数据空间中的抽象接入主体。

Demo 中至少包含两个典型连接器：

| Connector | 角色 | 说明 |
| --- | --- | --- |
| Connector A | 数据提供方 | 发布数据、定义访问策略、触发数据加密。 |
| Connector B | 数据使用方 | 请求访问数据、提交身份和属性、尝试解密。 |

Connector 职责：

- 生成身份密钥对。
- 向 CA 申请证书。
- 向 Platform 注册。
- 维护自身属性集合。
- 发起数据发布或数据访问请求。

### 4.3 CA

CA 是身份信任机构。

职责：

- 初始化根证书。
- 签发 Connector 证书。
- 管理证书状态。
- 提供证书验证接口。
- 提供证书撤销接口。

Demo 中 CA 可使用模拟证书对象，不强制生成真实 X.509 证书，但字段应接近真实证书。

### 4.4 Attribute Authority

属性授权机构负责属性可信和 ABE 相关能力。

职责：

- 初始化 CP-ABE 系统公钥和主密钥。
- 定义属性命名空间。
- 为 Connector 绑定属性。
- 根据 Connector 属性生成 ABE 属性私钥。
- 处理属性变更、属性撤销和属性密钥更新。

AA 持有 ABE master secret key。KMS 不直接生成 ABE 属性私钥。

### 4.5 KMS

密钥管理服务负责密钥生命周期。

职责：

- 初始化 KMS 主密钥。
- 生成数据加密密钥 DEK。
- 记录 DEK 元数据。
- 管理密钥版本和状态。
- 处理密钥轮换、撤销、禁用、销毁。
- 保存加密后的 DEK 记录。

KMS 第一版可以用内存或本地文件存储密钥材料，但必须在模型中标注密钥状态。

### 4.6 Crypto Engine

密码引擎封装具体算法。

职责：

- AES-GCM 加密和解密。
- CP-ABE setup、keygen、encrypt、decrypt。
- 预留 SM2、SM3、SM4 接口。

第一版 Demo 可使用演示型 CP-ABE：

- 真实执行策略解析和属性匹配。
- 使用可替换的密钥封装结构模拟 CP-ABE 密文。
- 接口保持与真实 CP-ABE 一致。

### 4.7 Data Resource

数据资源是被发布、加密和访问的对象。

资源由以下部分组成：

- 明文数据，演示时可短暂存在。
- AES-GCM 密文数据。
- CP-ABE 加密后的 DEK。
- 访问策略。
- 所属提供方 Connector。
- 密钥版本和资源状态。

## 5. 信任边界设计

### 5.1 逻辑信任边界

```text
┌────────────────────────可信控制域─────────────────────────┐
│ Platform                                                   │
│ CA                                                         │
│ Attribute Authority                                        │
│ KMS                                                        │
│ Crypto Engine                                              │
└───────────────────────────┬───────────────────────────────┘
                            │
┌───────────────────────────▼───────────────────────────────┐
│ Connector 模拟主体                                          │
│ Connector A / Connector B                                  │
└───────────────────────────────────────────────────────────┘
```

Demo 中所有模块运行在同一后端内，但逻辑上：

- CA 是身份信任根。
- AA 是属性信任根。
- KMS 是密钥状态信任根。
- Platform 是业务编排信任根。

### 5.2 数据可信假设

Demo 的可信假设如下：

1. CA 私钥未泄露。
2. AA 的 ABE master key 未泄露。
3. KMS master key 未泄露。
4. Platform 能正确执行注册和访问策略。
5. Connector 属性由 AA 授权，不由 Connector 自报。
6. 加密数据离开系统后仍保持密文状态。

## 6. 形式化模型

### 6.1 集合定义

定义系统实体集合：

```text
P = { platform }
CA = { ca }
AA = { attribute_authority }
KMS = { kms }
C = { c1, c2, ..., cn }              // Connector 集合
U = { u1, u2, ..., um }              // 用户或操作主体，可选
R = { r1, r2, ..., rk }              // 数据资源集合
A = { a1, a2, ..., aq }              // 属性全集
K = { k1, k2, ..., kt }              // 密钥集合
Cert = { cert1, cert2, ..., certn }  // 证书集合
Policy = { p1, p2, ..., ps }         // 访问策略集合
```

### 6.2 Connector 模型

每个 Connector 定义为：

```text
Connector c = (
  connector_id,
  name,
  role,
  public_key,
  private_key_ref,
  certificate_id,
  attributes,
  status,
  created_at
)
```

其中：

```text
role ∈ { PROVIDER, CONSUMER, BOTH }
status ∈ { PENDING, CERTIFIED, REGISTERED, DISABLED, REVOKED }
attributes ⊆ A
```

### 6.3 证书模型

证书定义为：

```text
Certificate cert = (
  certificate_id,
  subject_id,
  subject_name,
  public_key,
  issuer,
  serial_number,
  fingerprint,
  valid_from,
  valid_to,
  status
)
```

其中：

```text
status ∈ { VALID, EXPIRED, REVOKED }
```

证书验证函数：

```text
VerifyCert(cert, ca_public_key, now) -> Boolean
```

成立条件：

```text
cert.status = VALID
now ∈ [valid_from, valid_to]
cert.issuer = CA
cert.signature 可由 ca_public_key 验证
```

Demo 中可简化为字段校验和签名字符串校验。

### 6.4 属性模型

属性定义为：

```text
Attribute a = (
  name,
  value,
  namespace,
  issuer,
  status
)
```

属性实例可表示为：

```text
department=rd
role=researcher
level=3
project=alpha
```

属性状态：

```text
status ∈ { ACTIVE, REVOKED, EXPIRED }
```

Connector 的属性集合：

```text
Attr(c) = { a | a 被 AA 授权给 c 且 a.status = ACTIVE }
```

### 6.5 访问策略模型

访问策略定义为属性上的布尔表达式：

```text
policy p = BooleanExpression(A)
```

示例：

```text
department=rd AND role=researcher
department=sales OR level>=3
project=alpha AND role=auditor
```

第一版 Demo 可以先支持：

```text
AND
OR
括号
属性等值判断
```

策略满足函数：

```text
Satisfy(Attr(c), p) -> Boolean
```

当且仅当 Connector 的有效属性集合满足策略表达式时返回 true。

### 6.6 密钥模型

密钥定义为：

```text
Key k = (
  key_id,
  key_type,
  owner_id,
  algorithm,
  material_ref,
  status,
  version,
  usage,
  created_at,
  expires_at,
  parent_key_id,
  policy_id
)
```

密钥类型：

```text
key_type ∈ {
  KMS_MASTER_KEY,
  DATA_ENCRYPTION_KEY,
  ABE_PUBLIC_KEY,
  ABE_MASTER_SECRET_KEY,
  ABE_USER_SECRET_KEY,
  CONNECTOR_IDENTITY_KEY
}
```

密钥状态：

```text
status ∈ { ACTIVE, DISABLED, REVOKED, ROTATED, DESTROYED }
```

### 6.7 数据资源模型

数据资源定义为：

```text
DataResource r = (
  resource_id,
  name,
  provider_connector_id,
  ciphertext,
  encrypted_dek,
  encryption_algorithm,
  abe_policy,
  dek_key_id,
  key_version,
  status,
  created_at
)
```

资源状态：

```text
status ∈ { DRAFT, PUBLISHED, DISABLED, REVOKED, DESTROYED }
```

## 7. 核心安全关系

### 7.1 身份可信关系

Connector 可信需要满足：

```text
TrustedConnector(c) =
  c.status = REGISTERED
  AND Exists cert ∈ Cert:
      cert.subject_id = c.connector_id
      AND VerifyCert(cert, CA.public_key, now) = true
```

### 7.2 属性可信关系

Connector 属性可信需要满足：

```text
TrustedAttributes(c) =
  For all a ∈ Attr(c):
      a.issuer = AA
      AND a.status = ACTIVE
```

### 7.3 访问许可关系

Connector `c` 可以访问资源 `r`，需要满足：

```text
CanAccess(c, r) =
  TrustedConnector(c)
  AND TrustedAttributes(c)
  AND r.status = PUBLISHED
  AND KeyStatus(r.dek_key_id) = ACTIVE
  AND Satisfy(Attr(c), r.abe_policy)
```

### 7.4 解密成功关系

解密成功定义为：

```text
DecryptSuccess(c, r) =
  CanAccess(c, r)
  AND ABE_Decrypt(ABE_User_Key(c), r.encrypted_dek) = DEK
  AND AES_Decrypt(DEK, r.ciphertext) = plaintext
```

如果任一条件不满足，Demo 应展示失败原因：

- Connector 未注册。
- 证书无效或被撤销。
- 属性不满足策略。
- ABE 属性密钥被撤销。
- DEK 被撤销。
- 资源未发布或被撤销。

## 8. CP-ABE + AES 混合加密模型

### 8.1 加密侧责任划分

| 问题 | 设计答案 |
| --- | --- |
| 谁生成 AES 密钥？ | KMS 生成每份资源的随机 DEK。 |
| 谁保存 AES 密钥？ | 不保存明文 DEK；保存 DEK 元数据和 CP-ABE 加密后的 DEK。 |
| ABE 密钥在哪里生成？ | Attribute Authority 初始化 ABE 系统，并为 Connector 属性生成 ABE 用户密钥。 |
| 属性由谁管理？ | Attribute Authority 管理属性定义、绑定、更新和撤销。 |
| 连接器如何获得访问权限？ | 连接器先通过 CA/Platform 注册，再由 AA 授权属性；访问时属性需满足资源策略。 |
| 属性变化后如何更新权限？ | 旧属性私钥撤销，生成新属性私钥；必要时资源重新加密 DEK。 |

### 8.2 数据发布加密算法

输入：

```text
plaintext
provider_connector_id
access_policy
```

输出：

```text
DataResource
```

流程：

```text
1. Platform 验证 provider_connector_id 是可信 Connector。
2. KMS 生成 DEK:
   dek = GenerateRandom(256bit)
3. Crypto Engine 使用 AES-GCM 加密数据:
   ciphertext = AES_GCM_Encrypt(dek, plaintext)
4. Attribute Authority 获取 ABE public key。
5. Crypto Engine 使用 CP-ABE 加密 DEK:
   encrypted_dek = CPABE_Encrypt(abe_public_key, access_policy, dek)
6. KMS 记录 DEK 元数据:
   key_type = DATA_ENCRYPTION_KEY
   status = ACTIVE
   version = 1
7. Platform 创建 DataResource:
   ciphertext
   encrypted_dek
   abe_policy
   dek_key_id
   provider_connector_id
   status = PUBLISHED
```

### 8.3 数据访问解密算法

输入：

```text
consumer_connector_id
resource_id
```

输出：

```text
plaintext 或失败原因
```

流程：

```text
1. Platform 验证 consumer_connector_id 是可信 Connector。
2. Platform 查询资源 resource_id。
3. Platform 检查资源状态为 PUBLISHED。
4. KMS 检查资源关联 DEK 状态为 ACTIVE。
5. AA 查询 Connector 当前有效属性集合。
6. AA 判断属性集合是否满足 resource.abe_policy。
7. 若不满足，返回 ACCESS_DENIED。
8. AA 为 Connector 生成或获取 ABE_USER_SECRET_KEY。
9. Crypto Engine 使用 ABE 用户密钥解密 encrypted_dek，得到 DEK。
10. Crypto Engine 使用 DEK 执行 AES-GCM 解密，得到 plaintext。
11. Platform 记录访问日志。
12. 返回 plaintext 和流程状态。
```

## 11. 关键业务流程设计

### 11.1 系统初始化流程

```text
1. 用户点击“初始化系统”。
2. Platform 创建初始状态。
3. CA 生成 CA 根密钥和根证书。
4. Attribute Authority 生成 ABE public key 和 ABE master secret key。
5. KMS 生成 KMS master key。
6. 系统写入初始属性定义：
   - department=rd
   - department=sales
   - role=researcher
   - role=auditor
   - level=1
   - level=2
   - level=3
7. 返回系统状态：
   CA_READY
   AA_READY
   KMS_READY
   PLATFORM_READY
```

每一步的责任：

| 步骤 | 发起方 | 执行方 | 使用密钥 | 返回结果 |
| --- | --- | --- | --- | --- |
| 初始化 CA | 用户/API | CA | CA root key | Root CA 证书 |
| 初始化 AA | 用户/API | AA | ABE master key | ABE public key |
| 初始化 KMS | 用户/API | KMS | KMS master key | KMS 状态 |
| 初始化 Platform | 用户/API | Platform | 无 | 平台状态 |

### 11.2 Connector 注册认证流程

```text
1. Connector A 生成身份密钥对。
2. Connector A 向 CA 提交证书申请。
3. CA 验证申请信息。
4. CA 签发 Connector 证书。
5. Connector A 向 Platform 提交注册请求。
6. Platform 验证 Connector 证书。
7. Platform 创建 Connector 记录。
8. AA 根据注册请求绑定初始属性。
9. Platform 返回注册成功。
```

责任说明：

| 步骤 | 谁发起 | 谁验证 | 使用什么密钥 | 返回什么 |
| --- | --- | --- | --- | --- |
| 生成身份密钥 | Connector | Connector 本地 | Connector 私钥 | 公钥/私钥引用 |
| 申请证书 | Connector | CA | Connector 公钥，CA 私钥 | Connector 证书 |
| 平台注册 | Connector | Platform | CA 公钥验证证书 | Connector ID |
| 属性绑定 | Platform | AA | AA 内部信任 | 属性绑定记录 |

### 11.3 数据发布流程

```text
1. Connector A 发起数据发布请求。
2. Platform 验证 Connector A 已注册且证书有效。
3. Platform 校验访问策略语法。
4. KMS 生成 AES DEK。
5. Crypto Engine 使用 AES-GCM 加密明文数据。
6. AA 提供 ABE public key。
7. Crypto Engine 使用 CP-ABE 按策略加密 DEK。
8. KMS 保存 DEK 元数据，状态为 ACTIVE。
9. Platform 保存 DataResource。
10. 返回资源 ID、密文摘要、策略和流程步骤。
```

### 11.4 数据访问流程

```text
1. Connector B 请求访问 resource_id。
2. Platform 验证 Connector B 注册状态。
3. CA 验证 Connector B 证书状态。
4. Platform 查询资源状态。
5. KMS 检查资源关联 DEK 状态。
6. AA 查询 Connector B 当前属性。
7. AA 判断属性是否满足资源策略。
8. AA 生成或读取 Connector B 的 ABE_USER_SECRET_KEY。
9. Crypto Engine 使用 ABE 用户密钥解密 encrypted_dek。
10. Crypto Engine 使用 DEK 解密 ciphertext。
11. Platform 记录访问日志。
12. 返回明文或失败原因。
```

失败分支：

| 失败点 | 返回原因 |
| --- | --- |
| Connector 未注册 | `CONNECTOR_NOT_REGISTERED` |
| 证书无效 | `CERTIFICATE_INVALID` |
| 资源不存在 | `RESOURCE_NOT_FOUND` |
| 资源未发布 | `RESOURCE_NOT_PUBLISHED` |
| DEK 被撤销 | `DATA_KEY_REVOKED` |
| 属性不满足策略 | `POLICY_NOT_SATISFIED` |
| ABE 解密失败 | `ABE_DECRYPT_FAILED` |
| AES 解密失败 | `AES_DECRYPT_FAILED` |

### 11.5 属性更新流程

场景：

```text
Connector B:
department=rd -> department=sales
```

流程：

```text
1. 用户在 Connector 管理页面修改属性。
2. Platform 向 AA 提交属性更新请求。
3. AA 将旧属性绑定标记为 REVOKED。
4. AA 写入新属性绑定，状态为 ACTIVE。
5. AA 将旧 ABE_USER_SECRET_KEY 标记为 REVOKED。
6. AA 为新属性集合生成新的 ABE_USER_SECRET_KEY。
7. Platform 重新计算该 Connector 对已发布资源的可访问性。
8. 前端展示访问权限变化。
```

撤销强度：

| 模式 | 说明 | Demo 第一版 |
| --- | --- | --- |
| 软撤销 | 平台访问前检查属性和 key 状态，不允许旧属性访问。 | 默认采用。 |
| 强撤销 | 重新生成 DEK 并重新加密资源。 | 作为可选 rekey 操作展示。 |

### 11.6 密钥生命周期流程

创建：

```text
KMS.generate_dek(resource_id) -> KeyRecord(status=ACTIVE)
```

使用：

```text
DEK 仅在加密/解密运行时短暂出现。
明文 DEK 不在资源记录中保存。
```

轮换：

```text
1. KMS 生成新 DEK version+1。
2. Crypto Engine 用新 DEK 重新加密数据。
3. AA 用同一策略重新加密新 DEK。
4. 旧 DEK 标记为 ROTATED。
5. 资源指向新 key_version。
```

撤销：

```text
1. KMS 将 DEK 标记为 REVOKED。
2. Platform 拒绝新的访问请求。
3. 前端展示该资源不可解密。
```

销毁：

```text
1. KMS 将 key material 标记为 DESTROYED。
2. Demo 可保留元数据用于审计展示。
3. 后续访问返回 KEY_DESTROYED。
```

## 14. 状态机设计

### 14.1 Connector 状态机

```text
PENDING
  -> CERTIFIED
  -> REGISTERED
  -> DISABLED
  -> REVOKED
```

说明：

- `PENDING`：刚创建，未签发证书。
- `CERTIFIED`：已获得 CA 证书。
- `REGISTERED`：已在 Platform 注册。
- `DISABLED`：临时禁用。
- `REVOKED`：主体撤销，不可恢复。

### 14.2 Key 状态机

```text
ACTIVE
  -> DISABLED
  -> ACTIVE
  -> ROTATED
  -> REVOKED
  -> DESTROYED
```

说明：

- `DISABLED` 可恢复。
- `REVOKED` 不应用于新操作。
- `DESTROYED` 不能恢复，只保留审计元数据。

### 14.3 Resource 状态机

```text
DRAFT
  -> PUBLISHED
  -> DISABLED
  -> PUBLISHED
  -> REVOKED
  -> DESTROYED
```
